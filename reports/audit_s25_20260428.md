## Part 2 — Open Architectural Issues

### Issue A — `chunk_id` Namespace Collision (MEDIUM)

**Files:** `cogmem_api/engine/retain/orchestrator.py:134`, `cogmem_api/engine/retain/chunk_storage.py:21`

**What the plan specifies:**
> `chunk_id` format: `{bank_id}_{document_id}_{suffix}` where suffix is `p1_{chunk_index}` or `p2_{msg_idx}_{sub_idx}`. Two namespaces, no conflict.

**What is implemented:**
```python
# orchestrator.py:134
processed_fact.chunk_id = f"{bank_id}_{fact_doc_id}_{extracted_fact.chunk_index}"
```

`extracted_fact.chunk_index` is:
- Pass 1: `Pass1Chunk.chunk_index` — sequential integer (0, 1, 2…)
- Pass 2: `Pass2Chunk.chunk_index` which equals `msg_idx` — ALSO a small integer

**Collision scenario:** Messages = [user-0, assistant-1, user-2]:
- P1 chunk 0 → all P1 facts: `chunk_id = "{bank}_{doc}_0"`
- P2 chunk from msg_idx=0 → P2 facts from user-0: `chunk_id = "{bank}_{doc}_0"` ← **COLLISION**

The `chunk_id_suffix` field computed in `Pass1Chunk` (`p1_0`) and `Pass2Chunk` (`p2_0_0`) is never read anywhere. It was designed exactly for this namespace separation but is unused in both the orchestrator and the storage path.

**Impact:**
- P1 and P2 facts from index 0 share the same `chunk_id` in the DB, making them indistinguishable by chunk_id in queries.
- `chunk_storage.store_chunks_batch` no-op path: `{chunk.chunk_index: ... for chunk in chunks}` — when P1 chunk 0 and P2 chunk from msg 0 both appear in `chunks`, their `chunk_index=0` dict entry overwrites, silently losing one ChunkMetadata.
- Diagnostics and `diagnose_bank.py` that group by `chunk_id` will produce incorrect groupings.

**Fix:**
Add `chunk_id_suffix: str | None = None` to `ExtractedFact`. Propagate from chunk in `_extract_facts_with_llm`. Use in orchestrator:

```python
# In _extract_facts_with_llm (pass 1):
for pf in parsed:
    pf.raw_snippet = chunk.text
    pf.chunk_id_suffix = chunk.chunk_id_suffix  # "p1_0", "p1_1", etc.

# In _extract_facts_with_llm (pass 2):
for pf in filtered:
    pf.raw_snippet = chunk.text
    pf.chunk_id_suffix = chunk.chunk_id_suffix  # "p2_0_0", "p2_2_0", etc.

# In orchestrator.py:
suffix = getattr(extracted_fact, "chunk_id_suffix", None) or str(extracted_fact.chunk_index)
processed_fact.chunk_id = f"{bank_id}_{fact_doc_id}_{suffix}"
```

### Issue B — Causal Relation Indices Break After Dedup (MEDIUM)

**Files:** `cogmem_api/engine/retain/dedup.py`, `cogmem_api/engine/retain/link_creation.py`

**Root cause (pre-existing + S25 worsens):**

In `_normalize_llm_facts`, the LLM returns relations with `target_fact_index` values that are LOCAL to the single chunk's response (e.g., "fact 0 in this response causes fact 1 in this response"). However, `create_causal_links_batch` and `create_transition_links_batch` use these indices as GLOBAL indices into the full `unit_ids` list.

This pre-existing bug means relations from chunk N point to wrong facts when the chunk is not the first. S25 makes this worse in two ways:

1. **Dedup reorders facts:** P1 facts that survive dedup are followed by P2 facts. A P1 world/action_effect fact with `causal_relation.target_index=0` (pointing to the first fact in its LLM response) will point to a wrong fact in the global merged list.

2. **Missing P2 relations:** `extract_causal_links=False` for Pass 2, so even if P2 extracts a experience→intention relation that P1 captured, after dedup removes P1's versions, the relation is lost.

**Impact:** Causal, action_effect, and transition links in the knowledge graph may point to wrong facts or be silently dropped. The severity depends on how often Ministral-3B generates cross-fact relations — for short sessions, it's often zero.

**Fix options (complex):**
- Option A: After dedup, remap `target_fact_index` using a lookup table `{old_p1_index → new_merged_index}`. Drop relations whose target was deduped away.
- Option B: Only use relations from P2 facts (since P2 is canonical for personal types). P1 relations for personal-type facts are discarded after dedup anyway.
- Option C: Ignore for now — Ministral-3B rarely generates well-formed cross-fact relations; this has limited practical impact.

**Recommended:** Option A with a post-dedup remapping pass. Defer if E7 eval shows no regressions in causal/transition link quality.

## Part 5 — Prompt Quality Assessment

### Pass 1 Prompt (`cogmem_api/prompts/retain/pass1.py`)

**Strengths:**
- Clear 6-type taxonomy with worked examples for each
- "ONLY use 'experience' if the USER did something specific" — good guard against assistant-attributed experience
- `action_effect` triplet structure with required fields is explicit
- Habit trigger keywords listed (always, usually, every morning, etc.)
- Fulfilled vs. planning intention distinction is clear

**Weaknesses:**
- "This is Pass 1 of 2-pass extraction" context missing — model doesn't know Pass 2 handles user-side intensively, so it may over-extract user experiences in P1 (creating more dedup candidates)
- `RELATIONS` section at the bottom has no model-facing examples, making it hard for 3B-scale models to use correctly
- "Prioritize cross-turn synthesis facts" — not in prompt, meaning P1 treats all facts equally

### Pass 2 Prompt (`cogmem_api/prompts/retain/pass2.py`)

**Overall: Well-designed for the task.**

**Strengths:**
- "This is Pass 2" identifier is explicit (required for FakeLLM and model orientation)
- "BE AGGRESSIVE: even a brief statement like 'I just got this kit' MUST be extracted" — excellent instruction targeting exactly the LongMemEval miss scenario
- Allows only 4 personal types — model cannot return world or action_effect even by accident
- Entity emphasis with domain-specific examples (kit names, brands)
- `why` field for experience and intention — captures motivation context
- Examples directly address the motivating failure cases (Spitfire, Tiger I)

**Weaknesses / Improvement Opportunities:**
1. **No `when` guidance for recency markers.** When user says "I recently finished" or "last month", the `when` field should capture "recently" or "last month". Adding: `"when": "recently" | "last week" | specific date — use relative terms if no exact date` would improve temporal metadata.

2. **`who` field has no examples** — for single-speaker input, `who` is almost always "user". Could be dropped or simplified: `"who": omit unless another named person is the subject`.

3. **80-word limit for `what` may encourage padding.** For brief mentions ("I just got this kit"), the model might pad to fill the limit. Consider adding: "For brief facts, a single sentence is enough — do not pad."

4. **No dedupe instruction.** If the same fact appears twice in slightly different wording, the model might emit two facts. Consider: "Each distinct event → exactly ONE fact, even if mentioned twice."

### Judge Prompt (`cogmem_api/prompts/eval/judge.py`)

**Strengths:**
- Category-specific rubrics (temporal, knowledge-update, preference, abstention) with appropriate leniency (off-by-one tolerance for temporal counts)
- Consistent JSON output format across all categories
- "If response mentions outdated + updated info, still correct" for knowledge-update — reasonable

**No issues found.**

### Generation Prompt (`cogmem_api/prompts/eval/generate.py`)

Not reviewed in depth (not changed in S25, secondary to recall quality). Defer to next audit.

## Part 7 — Pre-Existing Issues Surfaced by S25

### 7.1 — Relation Index Is Already Global-Broken in Single-Pass

The `target_fact_index` in causal/transition relations from the LLM is chunk-local (index within one LLM response), but `create_causal_links_batch` and `create_transition_links_batch` treat it as a global index into `unit_ids`. In multi-chunk single-pass extraction (e.g., long content), this already produces wrong links. S25 makes it worse because dedup reorders facts.

This is **pre-S25** and affects all multi-chunk retain calls. Fix: scope the index lookup to the same `content_index` + `chunk_index` combination. Or disable cross-fact relations until this is fixed (since Ministral-3B rarely generates useful ones).

### 7.2 — `_derive_content_from_messages` vs. `_parse_legacy_text` Asymmetry

When messages are retained, `content` is set to `"[user]: ...\n\n[assistant]: ..."` (via `_derive_content_from_messages`). If extraction ever falls to heuristic fallback, `_extract_fallback_facts` uses `content.content` — which has the derived format. The fallback then produces `world` facts from the joined text, which is reasonable but loses the per-speaker structure. This is an edge case (only occurs when LLM is down).
